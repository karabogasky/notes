HTML Best Practices

Develop good habits

Having never coded before, you are about to develop your own style.

There is a preferred method of formatting your code. It makes life easier for yourself and others who must decipher it later.

Developing good habits now will help you avoid problems and bugs, both now and long-term, when you tackle more complex challenges.

Your professor is not intent on being picky; she is trying to save you from frustration.

Use a proper DOCTYPE

A DOCTYPE tells the web browser which version of HTML you are using.

The DOCTYPE we will use indicates our code is written in HTML5:

<!DOCTYPE html>

When the DOCTYPE is missing, the browser enters a state called quirks mode, which affects backward compatibility. Quirks mode may break your web page.

Declare a character set

A character set tells the web browser how letters, numbers and special characters should be displayed. There are many sets which include foreign letters and glyphs.

We will use UTF-8. The following tag should appear in the <head> of our page.

<meta charset="UTF-8" />

UTF-8 is the most dominant character encoding on the web and includes more than 1 million characters.

Close your tags

Unless an element is self-closing, always close your tags.

Bad: <p>example

Good: <p>example</p>

Do not use deprecated elements

While the Web supports backward compatibility, you should refrain from using tags which are no longer supported by the W3C specifications.

Write code in lowercase

Although HTML is not case-sensitive, it is an accepted practice to write elements, attributes and values using lowercase letters.

Writing in lowercase ensures backward compatibility with XHTML (a previous version of HTML).

It also helps make your code scannable, discernable from text.

<p class="intro">This is a sentence</p>

One exception: It is acceptable to use uppercase letters in alt or title values.

Always provide attribute values

Most attributes require a value. A few do not.

Add a value, even when not required.

<audio controls="controls"></audio>

Always quote attribute values

Quoting attribute values not only adds visual cues for easier-to-read code, it helps avoid bugs when working with CSS and Javascript.

<img src="photo.jpg" alt="Sebastian the Ibis" />

Comment your code

<!-- This is a comment. -->

Commenting your code helps you understand why you did what you did.

They also enable whomever inherits your code to:

  • identify parts of a page easily
  • understand the logic behind the code
  • explain what each part of the code is supposed to do

Comments and proper style will aid you page maintenance.

Avoid double dashes in comments

<!-- If you must use double dashes in a comment, use two single dashes separated by a space - - like so. -->

Since double dashes are used at the beginning and end of a comment, using them within may invalidate your code or introduce a bug.

Encode HTML entities

HTML entities (or 'escape codes') are special characters. They begin with an ampersand (&) and end with a semicolon (;).

These may be symbols or characters used in foreign languages.

For example: ‡ ♠ £ € ©

Many entities have a name and number. Modern browsers should recognize entity names. If the symbol appears broken, use the entity number instead.

Character Code offers an awesome reference for HTML entities. The W3C provides a comprehensive list, too.

Indent your code

HTML is not sensitive to white space.*

The use of whitespace displays the hierarchy of the code, which aids readability.

Indent code by two spaces for each level. This makes it easier to vertically 'match' block-level opening and closing tags.

If you use the TAB key for indenting, make sure you set Tab Length in your application's preferences to two spaces.

The amount of spacing you use between lines is a matter of preference. Find a pattern which suits you and remain consistent.

* ...but inline elements are affected by whitespace when certain styles applied. We will address this occurrance when we cover CSS.

Name your files properly

Always name the files for your web site using lowercase letters and no spaces. Avoid special characters.

While your computer may not see the difference between photo.jpg and photo.JPG, case-sensitive web servers sees these as different files.

Additionally, using spaces or special characters in filenames introduces hidden entities. A file named miami beach.jpg on your computer will be seen as miami%20beach.jpg on a web server.

Hyphens (-) or underscores (_) are preferable, though hyphens are better for SEO purposes.

Bad: Flynn's photo.jpg

Good: flynns-photo.jpg
